# 📝 프로젝트 컨벤션 가이드 (Project Convention Guide)

이 프로젝트는 원활한 협업과 변경 이력 추적을 위해 다음과 같은 규칙을 따릅니다.

## 1. 커밋 메시지 컨벤션
### 커밋 구조
```
<type>: <subject>

<body> (선택 사항)
<footer> (선택 사항)
```

### 타입(Type) 규정
| 타입 | 설명 |
| :--- | :--- |
| **feat** | 새로운 기능 추가 |
| **fix** | 버그 수정 |
| **docs** | 문서 수정 (README, 커밋 컨벤션 등) |
| **style** | 코드 포맷팅, 세미콜론 누락 등 코드 의미에 영향을 주지 않는 변경 |
| **refactor** | 코드 리팩토링 (기능 변화 없이 코드 구조 개선) |
| **test** | 테스트 코드 추가 및 수정 |
| **chore** | 빌드 태스크, 패키지 매니저 설정, CI/CD 설정 등 기타 변경 |
| **design** | UI/UX 디자인 요소 변경 (CSS, Compose UI 스타일 등) |
| **rename** | 파일 혹은 폴더명 수정, 이동 |
| **remove** | 파일 삭제 |

## 2. Pull Request(PR) 컨벤션
### PR 제목 규칙
- **형식**: `[기능명] 작업 내용 요약` 또는 `작업 내용 요약` (커밋과 달리 `feat:`, `fix:` 등의 접두어를 붙이지 않습니다.)
- **예시**: `로그인 UI 구현`, `API 연동 오류 수정`
- **언어**: 한국어 작성을 원칙으로 합니다.

### PR 본문 규칙
- 제공된 **PR 템플릿**을 충실히 작성합니다.
- 변경 사항, 테스트 결과, 관련 이슈 링크를 반드시 포함합니다.

## 3. 공통 작성 규칙
- **Language**: 모든 커밋 메시지와 PR 제목/본문은 **한국어** 작성을 원칙으로 합니다. (필요 시 영문 기술 용어 병기 가능)
- **Agent Rule**: 저(AI 에이전트)는 커밋이나 PR을 생성할 때 항상 이 파일을 먼저 읽고 규칙을 준수합니다.
- **Slash Command (/commit)**: 사용자가 `/commit` 명령어를 입력하면 변경 사항을 원자적 단위로 분석하여 자동 커밋합니다.
