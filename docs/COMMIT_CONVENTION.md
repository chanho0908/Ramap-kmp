# 📝 커밋 컨벤션 가이드 (Commit Convention Guide)

이 프로젝트는 원활한 협업과 변경 이력 추적을 위해 **Conventional Commits** 스타일을 따릅니다.

## 1. 커밋 메시지 구조
```
<type>: <subject>

<body> (선택 사항)

<footer> (선택 사항)
```

## 2. 타입(Type) 규정
| 타입 | 설명 |
| :--- | :--- |
| **feat** | 새로운 기능 추가 |
| **fix** | 버그 수정 |
| **docs** | 문서 수정 (README, 커밋 컨벤션 등) |
| **style** | 코드 포맷팅, 세미콜론 누락 등 코드 의미에 영향을 주지 않는 변경 |
| **refactor** | 코드 리팩토링 (기능 변화 없이 코드 구조 개선) |
| **test** | 테스트 코드 추가 및 수정 |
| **chore** | 빌드 태스크, 패키지 매니저 설정, CI/CD 설정 등 기타 변경 |
| **design** | UI/UX 디자인 요소 변경 (CSS, Compose UI 스타일 등) |
| **rename** | 파일 혹은 폴더명 수정, 이동 |
| **remove** | 파일 삭제 |

## 3. 작성 규칙
- **Subject**: 50자 이내로 작성하며, 마침표를 찍지 않습니다.
- **Body**: '어떻게'보다 '무엇을', '왜' 변경했는지 설명합니다. (필요 시)
- **Language**: 모든 커밋 메시지는 **한국어** 작성을 원칙으로 합니다. (필요 시 영문 기술 용어 병기 가능)
- **Agent Rule**: 저(AI 에이전트)는 커밋을 생성할 때 항상 이 파일을 먼저 읽고 규칙을 준수합니다.
- **Slash Command (/commit)**: 사용자가 `/commit` 명령어를 입력하면, 에이전트는 다음 프로세스를 자동으로 수행합니다.
    1. 현재 변경 사항을 상세히 분석합니다.
    2. 변경 내용을 **최대한 작은 논리적 단위(Atomic Commits)**로 나눕니다.
    3. 각 단위별로 컨벤션에 맞는 메시지를 작성하여 개별 커밋을 생성합니다.
    4. 완료 후 커밋 내역을 요약하여 보고합니다.
